# MirrorMind Progress 목적

## 배경

Codex 작업 환경이 점점 단일 에이전트 중심이 아니라, 여러 에이전트가 동시에 서로 다른 범위를 맡아 수행하는 방향으로 가고 있다. 이때 사용자는 개별 로그를 일일이 읽지 않고도 전체 상황을 빠르게 파악해야 한다.

이 페이지의 목적은 단순히 "진척률을 예쁘게 보여주는 것"이 아니다. 핵심은 여러 Codex 에이전트가 동시에 움직이는 상황에서, 사용자가 현재 상태를 이해하고 적절히 대응하도록 돕는 운영용 인터페이스를 만드는 것이다.

## 핵심 목적

이 페이지는 다음 질문에 빠르게 답해야 한다.

1. 지금 어떤 에이전트들이 살아 있는가.
2. 각 에이전트가 무엇을 맡고 있는가.
3. 어디까지 끝났고 무엇이 막혔는가.
4. 사용자 응답이나 의사결정이 필요한 지점이 어디인가.
5. 에이전트끼리 충돌하거나 중복 작업할 위험이 있는가.

즉 이 페이지의 목적은 `progress dashboard`라기보다 `다중 Codex 운영 상황판`에 가깝다.

## 사용자 관점의 가치

사용자는 이 페이지를 보고 다음 중 하나를 빠르게 결정할 수 있어야 한다.

- 지금 바로 개입해야 하는가.
- 잠시 기다려도 되는가.
- 방향을 바꿔야 하는가.
- 특정 에이전트에 추가 지시를 내려야 하는가.
- 충돌 위험이 있으니 작업 범위를 다시 조정해야 하는가.

핵심은 정보의 양 자체가 아니라, 대응 비용을 줄이는 것이다.

## 페이지가 보여줘야 할 정보

현재 기준으로 중요한 정보는 다음과 같다.

- 글로벌 진행상황
- 현재 활성 턴
- 최근 검증 상태
- 턴별 세부 진행상황
- 최근 사용자 요청과 최근 어시스턴트 응답

앞으로 더 중요해질 가능성이 큰 정보는 다음과 같다.

- 에이전트별 담당 범위
- 현재 상태: 실행 중 / 대기 / 완료 / 막힘
- `user action needed`
- 최근 결정 사항과 열린 질문
- 충돌 가능 파일 또는 작업 영역
- 마지막 활동 시각과 stale 여부

## 표현 원칙

표현은 화려한 대시보드보다 의미 구조를 우선한다.

- 문서처럼 읽혀야 한다.
- CSS를 꺼도 구조가 남아야 한다.
- `header`, `main`, `section`, `article`, `footer` 같은 시맨틱 구조가 먼저다.
- 시각 효과보다 상태 판단에 필요한 정보 배열이 중요하다.

## 한 줄 정의

MirrorMind Progress는 여러 Codex 에이전트가 동시에 일하는 환경에서, 사용자가 전체 상황을 인지하고 적절히 개입하도록 돕는 운영용 진행 문서다.
