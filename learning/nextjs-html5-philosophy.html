Next.js와 HTML5 철학의 관계

2026-04-02 세션115. HTML5 시맨틱 교재(html5-semantic-elements-study.md) 부록 A에서 분리. 핵심 질문: Next.js는 React가 HTML5 철학과 만든 긴장을 얼마나, 어떻게 해소하는가. 전제: HTML5 시맨틱 교재의 배경(3단계 변천사)과 6개 그룹을 이미 읽었다고 가정한다.

학습 방법: 퀴즈를 먼저 풀고, 해설을 읽는다.

배경: 긴장은 어디서 왔는가

HTML5 교재에서 정리한 핵심 긴장을 다시 짚는다. 이것이 이 문서 전체의 출발점이다.

HTML5의 3층 계약

HTML (의미) → CSS (표현) → JS (동작)
각 층이 실패해도 아래 층은 남는다

이것이 점진적 향상(Progressive Enhancement)이다. JS가 로드 안 되어도 콘텐츠는 읽힌다. CSS가 꺼져도 문서 구조는 남는다. 웹의 복원력(resilience)은 이 계약에서 나온다.

React SPA가 깬 것

React SPA:  JS → HTML → CSS
JS가 실패하면 빈 페이지

React가 HTML5 철학과 만든 3가지 긴장:

# 긴장 본질
1 점진적 향상 위반 JS가 HTML을 생성 — 3층 계약이 뒤집힘
2 div soup 문화 컴포넌트 추상화가 시맨틱을 가림
3 브라우저 기본 동작 재구현 <dialog> 대신 Modal 컴포넌트, <a> 대신 onClick

이 3가지 긴장을 기준으로 Next.js를 평가하는 것이 이 문서의 목적이다.

퀴즈: Next.js와 HTML5 철학

Q1. 아래 두 코드 중 어느 것이 HTML5 점진적 향상에 더 가까운가? 
// A: React SPA (Create React App)
// index.html
<body>
  <div id="root"></div>
  <script src="bundle.js"></script>
</body>

// B: Next.js App Router
// app/page.tsx (Server Component)
export default function Page() {
  return (
    <article>
      <h1>연구 노트</h1>
      <p>이 문서는 서버에서 HTML로 렌더링된다.</p>
    </article>
  )
}

정답: B.

A는 JS가 로드되기 전까지 빈 <div>만 존재한다. 검색 엔진, 스크린리더, 느린 네트워크 사용자가 보는 것은 빈 페이지다.

B는 서버에서 완성된 HTML이 만들어져서 내려온다. JS가 로드되기 전에도 <article>, <h1>, <p>가 브라우저에 존재한다. HTML5의 3층 계약이 복원된다.

Q2. Next.js의 Server Component는 왜 HTML5 철학에 가까운가?

Server Component는 서버에서 실행되고, 클라이언트에 JS를 보내지 않는다. 출력은 순수한 HTML이다.

// Server Component (기본값)
// 이 컴포넌트의 JS는 클라이언트 번들에 포함되지 않는다
export default async function ResearchList() {
  const papers = await db.papers.findMany()
  return (
    <section>
      <h2>최근 논문</h2>
      <ul>
        {papers.map(p => <li key={p.id}><a href={`/paper/${p.id}`}>{p.title}</a></li>)}
      </ul>
    </section>
  )
}

이 코드는 서버에서 실행된 후 클라이언트에는 <section>, <h2>, <ul>, <li>, <a> — 순수한 HTML만 도착한다. React SPA에서는 이 모든 것이 JS 번들 안에 있었다.

핵심: Server Component = HTML-first. JS가 필요한 곳(이벤트 핸들러, 상태)만 Client Component로 분리한다.

Q3. Next.js <Link>는 <a>인가 아닌가?

<a>이다.

import Link from 'next/link'

// 이것은:
<Link href="/about">소개</Link>

// 이렇게 렌더링된다:
<a href="/about">소개</a>

Next.js의 <Link>는 내부적으로 <a> 태그를 렌더링한다. 시맨틱이 유지된다. 스크린리더가 “링크”로 인식하고, 우클릭으로 새 탭에서 열 수 있고, 검색 엔진이 크롤링할 수 있다.

React Router의 <Link>도 <a>를 렌더링하지만, React SPA에서는 그 <a>가 JS 번들 안에 갇혀 있다. Next.js에서는 서버 렌더링된 HTML에 <a>가 직접 존재한다.

Q4. hydration 전에 버튼을 클릭하면 어떻게 되는가?

보이는데 안 눌린다.

서버 HTML 도착 (0ms)     → 콘텐츠가 보인다
                          → 링크(<a>)는 작동한다 (브라우저 기본 동작)
                          → 버튼(<button onClick>)은 보이지만 반응 없음
JS 번들 로드 (200ms~)    → hydration 시작
hydration 완료 (300ms~)  → 버튼 클릭이 작동한다

이 “보이는데 안 눌리는” 구간이 Next.js에서도 남아 있는 긴장이다. HTML5 점진적 향상의 이상 — “JS 없이도 기본 동작이 돌아간다” — 과 충돌한다.

해소 전략: - <a href>로 충분한 것은 <Link>를 쓴다 → hydration 불필요 - <form action>을 Server Action에 연결한다 → JS 없이도 폼이 제출된다 - 진짜 JS가 필요한 인터랙션만 Client Component로 격리한다

Q5. 다음 중 Next.js가 해소하지 못하는 긴장은? 
A. JS 로드 전 빈 페이지
B. div soup 문화
C. 브라우저 기본 동작 재구현 (일부)

정답: B와 C (일부).

  • A는 SSR/SSG/Server Components로 해소됨
  • B는 해소되지 않음 — 개발자가 <div>를 쓰면 <div>가 나온다. Next.js는 시맨틱을 강제하지 않는다
  • C는 부분 해소 — <Link>→<a>, <Image>→<img>, Server Actions→<form> 등 일부는 해소. 하지만 UI 라이브러리의 커스텀 모달, 데이트피커 등은 여전히 브라우저 기본 동작을 JS로 대체한다

긴장별 상세 분석

긴장 1: 점진적 향상 — 상당히 해소

React SPA의 가장 큰 문제는 3층 계약의 역전이었다. Next.js는 이것을 여러 메커니즘으로 복원한다.

SSR (Server-Side Rendering)

요청 → 서버에서 React 실행 → 완성된 HTML 응답 → 브라우저에 HTML 표시 → JS 로드 → hydration

JS가 로드되기 전에 HTML이 존재한다. 3층 계약이 복원된다.

SSG (Static Site Generation)

빌드 시 → React 실행 → HTML 파일 생성 → CDN 배포 → 브라우저에 HTML 표시 → JS 로드 → hydration

서버 비용이 없다. HTML 파일이 CDN에 있으므로 가장 빠르고, 가장 복원력이 높다.

Server Components (React 19 + Next.js App Router)

요청 → 서버에서 Server Component 실행 → HTML 스트리밍 → 
       Client Component만 JS 번들에 포함 → 부분 hydration

이것이 가장 HTML5 철학에 가까운 형태다:

특성 정적 HTML Server Component Client Component React SPA
HTML 선행 O O O (SSR) X
JS 번들 포함 X X O O
서버 자원 접근 X O X X
이벤트 핸들러 X X O O

Server Component는 서버에서 실행되고 HTML만 내보내는 컴포넌트다. async/await로 DB를 직접 읽고, 결과를 HTML로 스트리밍한다. 클라이언트에 JS를 안 보내므로 번들 크기가 줄고, hydration이 필요 없다.

핵심 통찰: Server Component는 “React로 작성하지만 출력은 정적 HTML”이라는 점에서, HTML5 철학의 현대적 구현이다.

Streaming

<!-- 전통 SSR: 전체 페이지가 준비될 때까지 기다림 -->
<!-- Streaming: 준비된 부분부터 보냄 -->

<html>
  <body>
    <header>사이트 헤더</header>           <!-- 즉시 도착 -->
    <main>
      <h1>논문 목록</h1>                   <!-- 즉시 도착 -->
      <!--$-->                             <!-- Suspense 경계 -->
      <template id="loading">로딩 중...</template>  <!-- 즉시 도착 -->
      <!--/$-->
      <!-- DB 쿼리 완료 후 실제 콘텐츠가 스트리밍으로 교체 -->
    </main>
  </body>
</html>

Streaming은 HTML을 점진적으로 보낸다. 느린 데이터 소스가 있어도 나머지 페이지는 먼저 도착한다. 이것은 HTML5의 복원력 원칙 — “가능한 한 복구해서 보여준다” — 과 같은 정신이다.

Server Actions — <form>의 복권

// Server Action
async function submitPaper(formData: FormData) {
  'use server'
  const title = formData.get('title')
  await db.papers.create({ data: { title } })
}

// Server Component
export default function NewPaper() {
  return (
    <form action={submitPaper}>
      <label htmlFor="title">논문 제목</label>
      <input id="title" name="title" type="text" required />
      <button type="submit">등록</button>
    </form>
  )
}

이 폼은 JS 없이도 작동한다. <form action>은 브라우저의 기본 동작이다. Server Action은 이 기본 동작을 서버 함수에 연결한다.

React SPA에서는:

// 전형적 React SPA 패턴 — HTML5 철학 위반
<form onSubmit={(e) => {
  e.preventDefault()  // 브라우저 기본 동작을 막는다!
  fetch('/api/papers', { method: 'POST', body: JSON.stringify(data) })
}}>

e.preventDefault()는 브라우저가 공짜로 주는 폼 제출을 JS로 가로채는 것이다. Server Action은 이 가로채기 없이 작동한다.

긴장 2: div soup — 해소되지 않음

Next.js는 시맨틱을 가능하게 하지만 강제하지 않는다.

// Next.js에서도 이렇게 쓸 수 있다 (그리고 많이 쓴다)
<div className="header">
  <div className="nav">
    <div className="item" onClick={() => router.push('/')}>홈</div>
  </div>
</div>

// HTML5 철학에 맞는 코드 (Next.js에서도 똑같이 쓸 수 있다)
<header>
  <nav>
    <ul><li><Link href="/">홈</Link></li></ul>
  </nav>
</header>

왜 해소되지 않는가:

  1. 컴포넌트 추상화가 시맨틱을 가린다. <Card />가 <div>인지 <article>인지 사용하는 쪽에서 모른다. UI 라이브러리(shadcn, Radix, MUI)의 내부 구현에 의존한다.

  2. Tailwind CSS 문화의 영향. Tailwind는 className에 스타일을 넣는다. 이것 자체는 문제가 아니지만, 요소 선택을 의식하지 않고 <div className="flex gap-4">를 기계적으로 쓰는 습관을 강화한다.

  3. Next.js의 자체 컴포넌트도 div를 래핑한다. <Link>와 <Image>는 시맨틱을 유지하지만, 레이아웃 관련 컴포넌트(layout.tsx의 구조)는 개발자 판단에 맡긴다.

대응: 이건 프레임워크가 풀 문제가 아니라 개발자의 판단 기준 문제다. HTML5 시맨틱 교재에서 배운 “이 요소가 무엇인가를 말하는가” 질문을 Next.js에서도 동일하게 적용해야 한다.

실용 지침: Next.js에서 시맨틱 유지하기

상황 div로 쓰기 쉬운 곳 시맨틱 대안
페이지 상단 영역 <div className="header"> <header>
네비게이션 <div className="nav"> <nav>
글 본문 <div className="post"> <article>
글 섹션 <div className="section"> <section> (제목 필수)
보조 정보 <div className="sidebar"> <aside>
페이지 하단 <div className="footer"> <footer>
검색 영역 <div className="search"> <search>
순수 레이아웃 <div className="flex"> <div> (의미 없는 레이아웃은 div가 맞다)

마지막 행이 중요하다. 모든 div가 나쁜 것이 아니다. 의미가 없는 순수 레이아웃 래퍼는 <div>가 정답이다. “이것이 무엇인가를 말할 수 있는데 div로 쓴 경우”만 문제다.

긴장 3: 브라우저 기본 동작 재구현 — 부분 해소

브라우저 기본 동작 React SPA Next.js 해소 정도
<a href> 네비게이션 React Router (JS 의존) <Link> → <a> 렌더링 + 프리패치 해소
<img> 이미지 <img> (최적화 없음) <Image> → <img> + srcset 자동 생성 + 최적화 래퍼 해소
<form> 제출 e.preventDefault() + fetch Server Actions → <form action> 해소
<meta> 메타데이터 react-helmet (JS 주입) metadata export (서버 렌더링) 해소
<details/summary> 접기 Accordion 라이브러리 여전히 라이브러리 의존 미해소
<dialog> 모달 Modal 라이브러리 여전히 라이브러리 의존 미해소
<input type="date"> DatePicker 라이브러리 여전히 라이브러리 의존 미해소

패턴: Next.js가 해소한 것은 프레임워크 수준의 기능(라우팅, 이미지, 폼, 메타데이터)이다. 해소하지 못한 것은 UI 컴포넌트 수준의 선택(모달, 아코디언, 날짜 선택기)이다. 후자는 개발자가 브라우저 네이티브 요소를 쓸지, 라이브러리를 쓸지 직접 판단해야 한다.

스펙트럼: 정밀 버전

HTML5 시맨틱 교재의 스펙트럼을 더 세밀하게 확장한다.

HTML5 철학에 가까움 ←──────────────────────────────────→ 멀어짐

정적     Next.js      Next.js      Next.js       React SPA
HTML     Server       SSR +        Client        (CSR only)
(순수)   Components   Server       Components    
         + SSG        Actions      + Hydration

각 단계의 특성:

단계 HTML 선행 JS 번들 서버 자원 인터랙션 HTML5 철학
정적 HTML O 없음 X X 완전 준수
SC + SSG O 없음 O (빌드 시) X 거의 준수
SSR + SA O 최소 O (요청 시) 폼 제출 가능 상당히 준수
CC + Hydration O 있음 X O 부분 준수
React SPA X 전부 X O 위반

핵심 통찰: Next.js App Router에서 기본값이 Server Component라는 점이 중요하다. 개발자가 의식적으로 'use client'를 선언해야 Client Component가 된다. 즉, 기본 경로가 HTML5 철학에 가까운 쪽이다. React SPA에서는 기본 경로가 반대쪽이었다.

Next.js에서도 남는 근본 긴장

Next.js가 많은 것을 해소했지만, 웹 프레임워크가 본질적으로 풀 수 없는 긴장이 있다.

1. hydration의 불쾌한 골짜기

[HTML 도착] ──── "보이는데 안 눌리는" 구간 ──── [hydration 완료]
             이 구간에서 사용자는 혼란을 느낀다

Server Component로 이 구간을 최소화할 수 있지만, Client Component가 있는 한 완전히 제거할 수 없다. 이것은 “서버에서 HTML을 보내되, 클라이언트에서 JS로 인터랙션을 붙인다”는 하이브리드 모델의 구조적 한계다.

2. 추상화가 시맨틱을 가리는 문제

// 이 코드를 읽는 사람은 최종 HTML을 모른다
<Card>
  <CardHeader>
    <CardTitle>연구 노트</CardTitle>
  </CardHeader>
  <CardContent>
    <p>내용</p>
  </CardContent>
</Card>

<Card>가 <article>인지 <div>인지는 라이브러리 내부 구현에 달려 있다. 이건 React의 컴포넌트 모델이 HTML 시맨틱과 만드는 구조적 긴장이다. Next.js가 풀 수 있는 문제가 아니다.

3. 생태계 관성

Next.js가 시맨틱을 가능하게 해도, npm 생태계의 대부분의 UI 라이브러리는 여전히 div 기반이다. 개발자가 라이브러리를 선택할 때 시맨틱 품질을 기준으로 삼는 문화가 형성되지 않으면, 프레임워크 수준의 개선은 한계가 있다.

실용 판단 기준: Next.js 프로젝트에서 HTML5 철학 적용

판단 질문 행동
Server vs Client 이 컴포넌트에 이벤트 핸들러나 상태가 필요한가? 아니면 Server Component (기본값)
시맨틱 요소 이 <div>가 무엇인가를 말할 수 있는가? 그렇다면 적절한 시맨틱 요소로 교체
네비게이션 router.push()를 쓰고 있는가? <Link href>로 교체 — JS 없이도 작동
폼 e.preventDefault()를 쓰고 있는가? Server Action + <form action>으로 교체
인터랙션 브라우저 네이티브로 가능한가? <details>, <dialog> 먼저 고려
CSS 끄기 테스트 CSS를 끄면 구조가 읽히는가? Next.js에서도 동일하게 적용

결론

Next.js는 React가 HTML5 철학과 만든 3가지 긴장 중 가장 큰 것(점진적 향상 위반)을 구조적으로 해소했다. Server Components는 “React로 작성하되 출력은 HTML-first”라는 새로운 균형점을 만들었다.

하지만 시맨틱은 프레임워크가 대신해주지 않는다. div를 쓰면 div가 나오고, <article>을 쓰면 <article>이 나온다. HTML5 시맨틱 교재에서 배운 판단 기준 — “이것이 무엇인가를 말하는가” — 은 Next.js에서도, 어떤 프레임워크에서도, 동일하게 적용되어야 한다.

Next.js가 해결한 것:  HTML이 먼저 도착하는가 (구조)
개발자가 해결할 것:   그 HTML이 의미를 담고 있는가 (판단)
