브라우저는 HTML을 어떻게 처리하는가

2026-04-02 세션115. HTML5 시맨틱 교재, Next.js 교재에 이은 3부작. 핵심 질문: URL을 입력한 순간부터 화면에 픽셀이 그려지기까지, 무슨 일이 일어나는가. 전제: HTML5 시맨틱 교재의 “의미 vs 표현” 구분, Next.js 교재의 SSR/hydration 개념을 알고 있다고 가정한다.

학습 방법: 퀴즈를 먼저 풀고, 해설을 읽는다.

전체 지도: URL → 픽셀

[네트워크]                    [파싱]                    [렌더링]
DNS → TCP → TLS → HTTP  →  HTML 파싱 → DOM 구축   →  렌더 트리
                             CSS 파싱 → CSSOM 구축      ↓
                             JS 실행 (블로킹!)        레이아웃
                                                       ↓
                                                     페인트
                                                       ↓
                                                     합성(Composite)
                                                       ↓
                                                     화면에 픽셀

이 문서는 왼쪽(네트워크)부터 오른쪽(픽셀)까지 순서대로 따라간다.

L3: 네트워크 — 요청이 서버에 닿기까지

퀴즈

Q1. 브라우저에 https://lighthouse.corca.ai를 입력하면, HTML이 도착하기 전에 몇 단계의 네트워크 핸드셰이크가 일어나는가?

정답 보기

최소 3단계: DNS → TCP → TLS

1. DNS 조회     "lighthouse.corca.ai의 IP가 뭐지?"  → 203.0.113.42
2. TCP 핸드셰이크  클라이언트↔서버 연결 수립 (SYN → SYN-ACK → ACK)
3. TLS 핸드셰이크  암호화 협상 (HTTPS이므로 필수)
4. HTTP 요청/응답  GET / HTTP/2 → 200 OK + HTML

HTML 바이트가 도착하기도 전에 여러 번의 왕복(round-trip)이 일어난다. 이것이 첫 바이트까지의 시간(TTFB, Time To First Byte)에 영향을 준다.

Q2. preconnect, prefetch, preload의 차이는?

<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="prefetch" href="/next-page.html">
<link rel="preload" href="/critical.css" as="style">
정답 보기
힌트 무엇을 하는가 언제 쓰는가 우선순위
preconnect DNS+TCP+TLS만 미리 수립 (다운로드 안 함) 곧 요청할 외부 도메인이 있을 때 높음
preload 리소스를 즉시 다운로드 (현재 페이지용) 브라우저가 늦게 발견하는 중요 리소스 높음 (필수)
prefetch 리소스를 유휴 시간에 다운로드 (다음 페이지용) 사용자가 곧 갈 것 같은 페이지 낮음

핵심: preload는 “지금 당장 필요하다”, prefetch는 “나중에 필요할 수 있다”, preconnect는 “연결만 미리 열어둬라”.

Next.js의 <Link>가 뷰포트에 보이면 자동으로 prefetch하는 것이 이 메커니즘이다.

각 단계 상세

1. DNS 조회 (Domain Name System)

브라우저: "lighthouse.corca.ai 의 IP 주소가 뭐지?"

[브라우저 캐시] → 없음
[OS 캐시]      → 없음  
[라우터 캐시]   → 없음
[ISP DNS 서버]  → 있음! → 203.0.113.42

도메인 이름을 IP 주소로 변환한다. 캐시에 있으면 즉시, 없으면 DNS 서버까지 왕복한다. 일반적으로 20~120ms.

2. TCP 핸드셰이크 (3-way)

클라이언트  ──SYN──→        서버
클라이언트  ←─SYN-ACK──    서버
클라이언트  ──ACK──→        서버
            [연결 수립됨]

신뢰할 수 있는 양방향 연결을 수립한다. 1 왕복(round-trip).

3. TLS 핸드셰이크

클라이언트  ──ClientHello──→        서버  (지원하는 암호화 방식 목록)
클라이언트  ←─ServerHello+인증서──  서버  (선택한 암호화 + 신원 증명)
클라이언트  ──키 교환──→            서버  (암호화 키 합의)
            [암호화 채널 수립됨]

HTTPS이면 필수. TLS 1.3에서는 1 왕복, TLS 1.2에서는 2 왕복.

4. HTTP 요청/응답

GET / HTTP/2
Host: lighthouse.corca.ai
Accept: text/html

→ 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 42381

<!DOCTYPE html>
<html>...

드디어 HTML 바이트가 도착하기 시작한다. 여기서부터 파싱이 시작된다.

HTTP/2와 HTTP/3

HTTP/1.1 HTTP/2 HTTP/3
연결당 요청 1개씩 순차 멀티플렉싱 (여러 요청 동시) 멀티플렉싱
헤더 텍스트 (매번 전송) 압축 (HPACK) 압축 (QPACK)
전송 프로토콜 TCP TCP QUIC (UDP 기반)
핸드셰이크 TCP + TLS 별도 TCP + TLS 별도 QUIC이 통합 (0-1 RTT)

HTTP/2의 멀티플렉싱이 중요한 이유: CSS, JS, 이미지를 하나의 연결에서 동시에 받을 수 있다. HTTP/1.1에서는 파일마다 별도 연결이 필요했다.

캐싱

[첫 방문]
브라우저 → 서버: GET /style.css
서버 → 브라우저: 200 OK + Cache-Control: max-age=31536000 + ETag: "abc123"
                 (1년간 캐시 가능)

[재방문]
브라우저: 캐시에 있고 만료 안 됨 → 서버에 요청하지 않음 (디스크 캐시 사용)

[캐시 만료 후]
브라우저 → 서버: GET /style.css + If-None-Match: "abc123"
서버: 내용 안 바뀜 → 304 Not Modified (본문 없이 응답, 대역폭 절약)

캐싱은 네트워크 비용을 극적으로 줄인다. Next.js의 SSG가 빠른 이유 중 하나 — 정적 HTML이 CDN에 캐시된다.

L1: 파싱과 트리 구축 — HTML이 구조가 되기까지

퀴즈

Q3. 다음 HTML이 브라우저에 도착했을 때, DOM 트리는 어떤 모양인가?

<html>
  <head><title>연구 노트</title></head>
  <body>
    <article>
      <h1>제목</h1>
      <p>본문</p>
    </article>
  </body>
</html>
정답 보기 
Document
└── html
    ├── head
    │   └── title
    │       └── "연구 노트" (텍스트 노드)
    └── body
        └── article
            ├── h1
            │   └── "제목" (텍스트 노드)
            └── p
                └── "본문" (텍스트 노드)

DOM(Document Object Model)은 HTML의 트리 형태 인메모리 표현이다. 브라우저가 HTML 바이트를 읽으면서 실시간으로 이 트리를 구축한다. JavaScript의 document.querySelector('article')은 이 트리의 노드를 찾는 것이다.

HTML5 교재와의 연결: <article>을 쓰면 DOM 트리에 article 노드가 생기고, 스크린리더와 검색 엔진이 이 노드를 의미적으로 해석한다. <div>를 쓰면 div 노드가 생기지만 의미가 없다. DOM이 시맨틱의 실체다.

Q4. CSS는 DOM 구축을 블로킹하는가, 렌더링을 블로킹하는가?

정답 보기

렌더링을 블로킹한다. DOM 구축은 블로킹하지 않는다.

HTML 파싱 ──→ DOM 구축 (CSS와 무관하게 진행)
CSS 파싱  ──→ CSSOM 구축 (병렬로 진행)

DOM 완성 ──┐
            ├──→ 렌더 트리 생성 (둘 다 필요)
CSSOM 완성 ┘

CSS가 아직 로드 중이면: - DOM은 계속 구축된다 (HTML 파싱은 멈추지 않는다) - 하지만 화면에 아무것도 그리지 않는다 (CSSOM이 완성될 때까지)

이것이 “CSS는 render-blocking”이라고 하는 이유다. CSS 규칙은 뒤에서 덮어쓸 수 있으므로(h1 { color: red } → h1 { color: blue }), 전부 파싱하기 전에는 최종 스타일을 확정할 수 없다.

HTML 파싱 → DOM

브라우저는 HTML 바이트를 점진적으로 파싱한다. 전체 문서가 도착할 때까지 기다리지 않는다.

바이트 → 문자 → 토큰 → 노드 → DOM 트리

"<article>"  →  시작 토큰 {tag: "article"}  →  Element 노드 생성
"제목"        →  텍스트 토큰                  →  Text 노드 생성  
"</article>" →  종료 토큰                    →  트리 위치 확정

점진적 파싱이 중요한 이유: Next.js Streaming은 이 특성을 활용한다. HTML을 조각씩 보내면, 브라우저는 도착하는 즉시 파싱하고 DOM에 추가한다. 전체를 기다리지 않아도 사용자에게 콘텐츠를 보여줄 수 있다.

CSS 파싱 → CSSOM

바이트 → 문자 → 토큰 → 노드 → CSSOM 트리

CSSOM:
├── html { font-size: 16px }
│   ├── body { margin: 0 }
│   │   ├── article { max-width: 36em }
│   │   │   ├── h1 { font-size: 2em; font-weight: bold }
│   │   │   └── p { line-height: 1.6 }

CSS는 상속과 캐스케이드 때문에 점진적으로 적용할 수 없다. 전체 CSS가 파싱되어야 최종 스타일이 확정된다. 이것이 CSS가 render-blocking인 이유다.

렌더 트리 = DOM + CSSOM

DOM                    CSSOM                   렌더 트리
├── html               html {font: 16px}       ├── html {font: 16px}
│   ├── head           head (display: none)    │   └── body {margin: 0}
│   │   └── title      ...                     │       └── article {max-width: 36em}
│   └── body           body {margin: 0}        │           ├── h1 {font-size: 2em}
│       └── article    article {max-width}     │           └── p {line-height: 1.6}
│           ├── h1     h1 {font-size: 2em}
│           └── p      p {line-height: 1.6}

렌더 트리에는 보이는 요소만 포함된다: - <head>, <title> → 포함 안 됨 (화면에 안 보임) - display: none 요소 → 포함 안 됨 - visibility: hidden 요소 → 포함됨 (공간은 차지)

레이아웃 (Reflow)

렌더 트리의 각 노드에 정확한 위치와 크기를 계산한다.

article { max-width: 36em }
→ 뷰포트 너비 1024px, 폰트 16px이면
→ 36em = 576px
→ article 박스: x=224, y=50, width=576, height=?

h1 { font-size: 2em }  
→ 2em = 32px
→ "제목" 텍스트가 한 줄에 들어감
→ h1 박스: x=224, y=50, width=576, height=44

p { line-height: 1.6 }
→ "본문" 텍스트
→ p 박스: x=224, y=94, width=576, height=26

왜 이것이 비용이 큰가: DOM이 변경될 때마다 레이아웃을 다시 계산해야 한다(reflow). 이것이 DOM 조작이 “비싸다”고 하는 이유 중 하나다. React의 Virtual DOM은 실제 DOM 변경을 최소화하여 reflow를 줄이는 전략이다.

페인트 (Paint)

레이아웃이 확정되면, 각 노드를 실제 픽셀로 그린다.

h1 "제목" → 폰트 로드 → 글리프 래스터라이즈 → 비트맵 생성
article 배경 → 배경색 채우기
border, shadow → 추가 그리기 명령

페인트는 레이어 단위로 일어난다. transform, opacity, will-change 속성이 있는 요소는 별도 레이어로 승격된다.

합성 (Composite)

여러 레이어를 최종 화면으로 합친다. GPU가 담당한다.

레이어 1: 배경
레이어 2: 본문 콘텐츠
레이어 3: 고정 헤더 (position: fixed)
레이어 4: 애니메이션 중인 요소

GPU: 레이어들을 z-order대로 겹쳐서 최종 프레임 생성

transform과 opacity 애니메이션이 빠른 이유: 레이아웃과 페인트를 건너뛰고 합성만 다시 하면 된다. left/top 애니메이션은 매 프레임 레이아웃부터 다시 해야 한다.

L2: JavaScript가 파이프라인을 막는 곳

퀴즈

Q5. 다음 세 가지 <script> 중 DOM 파싱을 멈추는 것은?

<script src="a.js"></script>
<script src="b.js" async></script>
<script src="c.js" defer></script>
정답 보기

a.js만 DOM 파싱을 멈춘다.

HTML 파싱:  ████████░░░░░░░████████████████
a.js:               ↓다운로드↓실행↓
                    파싱 중단!

HTML 파싱:  ████████████████████████████████
b.js (async):    ↓다운로드↓실행↓  (파싱과 병렬, 다운로드 완료 시 즉시 실행)

HTML 파싱:  ████████████████████████████████
c.js (defer):    ↓다운로드────↓              ↓실행 (파싱 완료 후)
속성 다운로드 실행 시점 파싱 블로킹 순서 보장
(없음) 파싱 중단하고 다운로드 다운로드 완료 즉시 O O
async 병렬 다운로드 다운로드 완료 즉시 실행 시에만 X
defer 병렬 다운로드 DOM 파싱 완료 후 X O

defer가 가장 안전하다 — 파싱을 안 막고, 순서도 보장하고, DOMContentLoaded 직전에 실행된다.

Q6. DOMContentLoaded와 load 이벤트의 차이는?

정답 보기 
HTML 파싱 완료 + defer 스크립트 실행
        ↓
  DOMContentLoaded 발생  ← "DOM 준비됨, 이미지는 아직"
        ↓
이미지, 폰트, iframe 등 모든 리소스 로드 완료
        ↓
      load 발생          ← "페이지 완전히 로드됨"
이벤트 언제 발생 기다리는 것 용도
DOMContentLoaded DOM 파싱 완료 시 HTML + defer JS DOM 조작 시작
load 모든 리소스 로드 완료 시 이미지, 폰트, CSS 등 전부 전체 페이지 준비 확인

대부분의 JS 초기화는 DOMContentLoaded에서 하면 된다. load를 기다리면 이미지 로딩 때문에 불필요하게 늦어진다.

Q7. React hydration은 이 파이프라인의 어디에 위치하는가?

정답 보기 
[서버]
Server Component 실행 → HTML 생성 → 클라이언트에 전송

[브라우저]
1. HTML 도착 → DOM 구축 (서버가 보낸 HTML 파싱)
2. CSS 도착 → CSSOM 구축
3. 렌더 트리 → 레이아웃 → 페인트 → 화면에 콘텐츠 표시 ← 여기서 "보인다"
4. JS 번들 도착 → 파싱 + 실행
5. React hydration 시작                              ← 여기서 "눌린다"
   - 기존 DOM 트리를 순회
   - 이벤트 핸들러 연결 (onClick, onChange 등)
   - 상태 초기화
6. hydration 완료 → 인터랙션 가능

3과 5 사이가 “보이는데 안 눌리는” 구간이다.

이 구간이 존재하는 이유: - 3단계(페인트)는 HTML+CSS만 있으면 된다 → 빠르다 - 5단계(hydration)는 JS 번들 다운로드 + 파싱 + 실행이 필요 → 느리다

Server Component가 이 구간을 줄이는 이유: Server Component는 JS 번들에 포함되지 않으므로, hydration 대상이 아니다. Client Component만 hydration하면 되므로 JS 번들이 작아지고, hydration이 빨라진다.

JS가 파싱을 블로킹하는 이유

JavaScript가 DOM 파싱을 중단시키는 이유는 단순하다: JS가 DOM을 변경할 수 있기 때문이다.

<p id="greeting">안녕하세요</p>
<script>
  document.getElementById('greeting').textContent = '반갑습니다'
</script>
<p>다음 문단</p>

브라우저는 <script>를 만나면 “이 스크립트가 위의 DOM을 바꿀 수도 있으니, 실행이 끝날 때까지 아래 HTML 파싱을 멈추자”고 판단한다. 실제로 위 코드에서 스크립트는 <p>의 텍스트를 바꾼다.

CSS도 JS 실행을 블로킹한다:

HTML 파싱 중 <script> 발견
→ 위에 아직 로드 안 된 CSS가 있는가?
→ 있다면: CSS 로드 완료까지 JS 실행도 대기
→ 이유: JS가 getComputedStyle()로 스타일을 읽을 수 있기 때문

이것이 “CSS는 render-blocking이고, JS는 parser-blocking이며, CSS는 JS를 간접적으로 parser-blocking한다”는 연쇄 블로킹의 전체 그림이다.

블로킹 해소 전략 요약

전략 무엇을 하는가 Next.js에서
<script defer> JS 다운로드는 병렬, 실행은 파싱 후 Next.js가 자동 적용
<script async> 다운로드·실행 모두 병렬 (순서 보장 안 됨) 분석 스크립트 등에 사용
CSS를 <head>에 파싱 초반에 CSS 다운로드 시작 Next.js가 자동 적용
Critical CSS 인라인 CSSOM 구축 대기 시간 제거 선택적 적용
Server Components JS 번들 자체를 안 보냄 App Router 기본값
Streaming HTML을 점진적으로 보냄 Suspense와 함께 자동

전체 연결: 3부작의 관계

[1. HTML5 시맨틱 교재]
   "이 요소가 무엇인가를 말하는가?"
   → DOM 트리에 의미 있는 노드를 만든다
   → 스크린리더, 검색 엔진이 이 트리를 해석한다

[2. Next.js와 HTML5 철학]
   "HTML이 먼저 도착하는가?"
   → SSR/SC는 서버에서 DOM을 구축하여 HTML로 보낸다
   → hydration은 기존 DOM에 이벤트를 연결하는 것이다

[3. 이 문서 — 브라우저 렌더링 파이프라인]
   "HTML이 어떻게 픽셀이 되는가?"
   → DOM + CSSOM → 렌더 트리 → 레이아웃 → 페인트 → 합성
   → JS가 이 과정을 어디서 어떻게 블로킹하는가
   → 네트워크 비용이 왜 중요한가

이 3편을 관통하는 하나의 질문:

“웹 페이지의 목적은 사용자에게 의미 있는 콘텐츠를 가능한 빨리, 가능한 안정적으로 전달하는 것이다. HTML5의 시맨틱, Next.js의 서버 렌더링, 브라우저의 점진적 파싱 — 이 모든 것은 같은 목적을 향한 다른 계층의 해법이다.”

용어 정리

용어 뜻 이 문서에서의 맥락
DOM Document Object Model — HTML의 트리 형태 인메모리 표현 파싱 결과물. JS가 조작하는 대상
CSSOM CSS Object Model — CSS의 트리 형태 표현 render-blocking의 원인
렌더 트리 DOM + CSSOM 결합. 보이는 요소만 포함 레이아웃의 입력
레이아웃 (Reflow) 각 요소의 위치와 크기 계산 DOM 변경 시 재계산 필요
페인트 레이아웃 결과를 픽셀로 변환 레이어 단위로 수행
합성 (Composite) 레이어들을 최종 화면으로 병합 GPU가 담당
TTFB Time To First Byte — 첫 바이트 도착 시간 네트워크 성능 지표
CRP Critical Rendering Path — 첫 화면까지의 최소 경로 최적화 대상
hydration 서버 렌더링된 HTML에 JS 이벤트를 연결하는 과정 “보이는데 안 눌리는” 구간의 원인
